# 사전원가 시뮬레이션 웹 시스템 (Pre-Cost Simulation)

제지/위생용지 공장의 월 사전원가를 산출·시뮬레이션하는 웹 시스템입니다.
**Wireframe v2.0** 기반의 클릭 가능한 프론트엔드 프로토타입(SPA)입니다.

## 프로젝트 개요
- **목표**: SAP BW로 수신한 기준정보 + 사용자 실적입력으로 사전원가를 계산하고, 변수 슬라이더로 실시간 시뮬레이션
- **디자인**: PPTX에서 추출한 Slate + Blue 엔터프라이즈 대시보드 디자인 시스템
- **산식**: module-calc(Java)를 JS로 포팅 — V·P·W·X·Y·Z·AA

## 산식 엔진 (calc.js)
- **V** 가중평균단가 = (R·S + T·U) / (R + T)
- **P** 사용량 = Q × 생산톤
- **W** 재료비(백만원) = P × V / 1,000,000
- **X** 톤당원가 = W / 생산톤 × 1,000,000
- **Y** 물량차이 = (L − Q) × M × 생산톤
- **Z** 가격차이 = Q × (M − V) × 생산톤
- **AA** 총차이 = Y + Z

### 검증 케이스 (LATEX 2000041, 4월 BW → 5월 사전원가)
입력: L=5.3643, M=1512.90, Q=5.20, V=1550, 생산톤=16,000
- Y = (5.3643 − 5.20) × 1512.90 × 16000 = **3,977,112** ✓
- Z = 5.20 × (1512.90 − 1550) × 16000 = **−3,086,720** ✓
- AA = Y + Z = **890,392** ✓

## 데이터 모드 (DEMO_MODE) — 데모 vs 운영 분리

현재 화면에 들어있는 데이터는 **모두 가상(데모) 데이터**입니다. 운영 서버 배포 시에는
가상 데이터 없이 **빈 상태(Empty State)**로 시작하도록 `public/static/data.js`에서 분리했습니다.

### 동작 방식
- **DEMO_MODE 자동 판별** (`data.js`):
  - `?demo=1` 쿼리 → 강제 데모(가상 데이터 표시)
  - `?demo=0` 쿼리 → 강제 운영(빈 상태)
  - 그 외: 호스트명이 `localhost`/`127.0.0.1`/`sandbox`/`e2b`/`novita` 면 데모, 운영 도메인이면 빈 상태
- **데모 모드**: `DEMO_DATA`(가상 데이터)를 복사해 `DB`에 채움 → 프로토타입 시연 그대로 표시
- **운영 모드**: `emptyDB()`(빈 배열 + `-` placeholder)로 시작 → 각 화면에 **"데이터 없음 / API 연동 대기"** Empty State 표시

### API 연동 (Spring Boot)
운영에서는 Spring Boot API 결과로 빈 `DB`를 채웁니다:
```js
fetch('/api/precost?plant=PM2&month=2026-05')
  .then(r => r.json())
  .then(payload => { DB.load(payload); route(); });
```
- `DB.load(payload)` : `meta/materials/movements/plans/bwRows/approvals/trend/alerts` 키를 주입하는 훅
- `DB.hasData()` : 데이터 유무 판별 (각 PAGE 함수가 이 값으로 Empty State 분기)

### 검증
- `?demo=1` → KPI/차트/표 정상 표시 (콘솔 에러 0)
- `?demo=0` → 전 화면 Empty State 정상 표시 (콘솔 에러 0, division-by-undefined 방지)

## ⚡ 전력비 시뮬레이션 모듈 (엑셀 05.이동계획 기반)

업로드된 엑셀 `05.이동계획`의 산식을 그대로 재현한 **실적 기반 전력비 시뮬레이터**입니다.

### 데이터
- **호기 13개**: 제지2·3, 화장지3·4·5, 가공2~6, 생리대5·6, 라미네이팅
- **기간**: 2026년 1~12월 (호기별 전력사용량/생산량/전력단가 실적)
- 데이터: `public/static/power-data.js` (`POWER_DEMO`) — 엑셀에서 추출, 검증 불일치 0건

### 산식 (호기 유형별 분기)
- **제지·화장지 (paper, 생산량=kg)**
  - 전력원단위[kWh/ton] = 전력사용량 ÷ 생산량 × 1000
  - 전력비원단위[천원/ton] = 전력원단위 × 전력단가 ÷ 1000
- **가공·생리대·라미네이팅 (proc, 생산량=EA)**
  - 전력원단위[kWh/개] = 전력사용량 ÷ 생산량
  - 전력비원단위[원/개] = 전력원단위 × 전력단가
- 엔진: `public/static/power-calc.js` (`PowerCalc.computeMonth`)

### 시뮬레이션 동작 (실적 기반 + 변수 조정) — 접근 1-A
- **생산량·가동시간 입력은 `예상 생산량 기준정보(#prodplan)` 탭에서만** 한다. 전력비 화면은 그 값을 **자동 반영(읽기 전용)**.
- **전력비 화면 입력(변수)**: 대상 월, 전력단가
- 입력값(예상 생산량·예상 가동시간)이 바뀌면 → 전력사용량·전력원단위·전력비원단위가 **실시간 재계산**
- 결과: 호기별 표(가동시간·전력사용량·생산량·원단위·전력비) + KPI(총 전력비 / 총 사용량 / 제지·화장지 톤당 전력비)
- 화면 상단에 `예상값 N개 호기 적용 중` 표시 + 기준정보 편집 바로가기 버튼 제공

### 전력사용량 예측 — A방식(가동시간 비례)
```
시간당전력[kWh/Hr] = 실적 전력사용량 ÷ 실적 가동시간      (실적 고정 계수)
예상 전력사용량   = 시간당전력 × 예상 가동시간(입력)
전력원단위        = 예상 전력사용량 ÷ 예상 생산량 (paper면 ×1000)
전력비원단위      = 전력원단위 × 전력단가
```
- **B방식 fallback**: 실적 가동시간이 없는 월/호기는 실적 전력사용량을 그대로 사용(전력비 표에 `B` 배지 표시).
- 검증: 제지3 2026-01 가동시간 +20% → 전력사용량 14,712,533 → 17,655,040 kWh(정확히 +20%), 원단위 567.65 → 681.18 (node 테스트 PASS).
- 실적 가동시간은 1~5월 데모에 반영(사용자 제공). 6~12월은 미입력 → B방식.

### 전력사용량 세분화 — 모고객 + ESS (NEW)
전력사용량(kWh)을 **① 모고객(설비/호기별)** 과 **② ESS(공장 단위)** 로 분리하여 계산한다.
```
전력사용량 = 모고객(usageMo) + ESS(usageEss)

① 모고객 = moBase + moCommon + moLami
   · moBase   : 호기별 A방식 — 시간당 모고객전력(실적) × 예상 가동시간
   · moCommon : 공통설비 배분 = (TOC + EDI + 복합보일러NOX + 에너지재활용) × 생산량비율[호기]
                - 폐수/스팀 비율 = 호기 생산량 / 전체 생산량 합. **현재 공통설비값 전부 0**(월초 환경에너지팀 확정).
                - moBase와 별도 추가(중복 아님). 라미네이팅 호기에는 배분하지 않음.
   · moLami   : 라미네이팅 고정값 — 제지2·제지3 가동(이동계획 prod>0) 판정
                제지2 & 제지3 모두 가동 → 23,000 / 하나만 → 15,000 / 둘 다 0 → 0
② ESS    : 충전식 — ESS충전보증량[호기] × (평일 + 토요일) × 효율(0.9)
```
- 공통설비가 현재 0이므로 **숫자는 기존(모고객 단순 A방식)과 동일**하며, 환경팀이 값을 입력하면 즉시 생산량 비율로 배분된다(`common` override / `overrides.common`).
- 라미네이팅 검증: 1~2월(제지2 미가동) = 15,000 / 3~12월(제지2·3 동시) = 23,000 — 데모 12/12 정확 일치.
- `#power` 표의 모고객 셀/합계 셀에 마우스 오버 → `A방식 / 공통설비배분 / 라미고정` 구성 tooltip 표시. 라미 호기는 `고정` 배지.
- **ESS충전보증량**: 호기별 고정값(kWh). 현재값은 **2026-01 실적에서 역산**하여 주입 (`essGuarantee`, power-data.js). 역산오차 ~1e-17.
- **일자(days)**: 충전 가동일 = 평일 + 토요일. 토요일은 **휴일(대체)** 로 취급. `#prodplan` 탭에서 평일/토요일/휴일 입력 가능(예상월만, 실적월 잠금).
- 검증: 제지3 2026-01 → 모고객 **14,135,440** + ESS **577,093** = 14,712,533 kWh (엑셀 E5/E6와 정확히 일치, node PASS).

### 전력요금 6항목 + 회계비용 (NEW)
전력요금을 6개 항목으로 구분하여 표시하고, 회계비용을 집계한다 (공장 전체 레벨, `byMonth[m].fee`).
```
①모고객요금(mo) ②ESS요금(ess) ③SPC지급금(spc) ④DR정산금(dr) ⑤복합보일러차감비(boiler) ⑥삼성보상금(samsung)

회계비용 = ①모고객 + ②ESS + ③SPC − ④DR − ⑤복합보일러차감비 − ⑥삼성보상금
           (※ SPC는 가산(+))
```
- 검증: 2026-01 회계비용 = **3,891,596,707 원** (엑셀 N열과 정확히 일치, node PASS).
- **요금 산식(요금 = 사용량 × 단가 …) 자체는 추후 반영 예정** — 현재는 SAP/이동계획 수신값을 표시·집계만 한다.
- `#power` 화면 하단 **전력요금 구성 카드**에 6항목 부호(±)와 회계비용 합계 표시.
- 실적월·예상월 **동일 산식**으로 계산.

### 실적/예상 기준일 (Cutoff) — 현재월 기준 자동 분할
현재일 기준으로 월을 **실적 확정월**과 **예상월**로 자동 분할한다. (`Period` 모듈, app.js)
```
기준월(cutoff) = 현재월(YYYY-MM)
month <  cutoff  → 실적 확정월  (SAP 실적, 입력·수정·단가조정 금지 / 잠금)
month >= cutoff  → 예상월       (이동계획 입력 + A방식 예측 적용)
```
- 예) **2026-05-31 기준 → 1~4월 실적(잠금), 5월~ 예상.** (사용자 예시와 일치, node 검증 PASS)
- **과거 실적 수정 금지**: 실적월은 `#prodplan` 입력칸·엑셀 업로드·`이 달 전체 실적값으로` 버튼 모두 비활성, `#power`는 단가 입력 비활성 + ProdPlan 예상값을 적용하지 않고 SAP 실적 그대로 표시. `ProdPlan.bulkSet`도 실적월 행은 자동 건너뜀.
- 월 선택 드롭다운에 `· 실적` / `· 예상` 라벨, 화면 상단에 기간 배너 표시. 기본 표시 월 = 기준월(데이터에 있으면).
- **데모/검증용 기준월 덮어쓰기**: URL에 `?asof=YYYY-MM` (예: `?asof=2026-08`) → 실제 today와 무관하게 해당 월을 기준월로 사용.

## 🏭 예상 생산량 기준정보 (#prodplan) — 전 시뮬레이션 공통 입력

생산량·가동시간은 모든 원가 모듈(전력비, 추후 재료비/구매비 등)에 영향을 주는 **공통 입력값**이므로,
별도의 기준정보 탭에서 한 번만 입력하고 각 모듈이 이를 공유한다. (사용자 확정 접근 **1-A**)

### 입력 구조 — 호기당 3행
호기마다 다음 **3개 항목**을 한 묶음으로 본다.
| 항목 | 입력 방식 | 산식 |
|---|---|---|
| ① 생산량 | 현업 직접 입력 | — (단위: 호기별 상이) |
| ② 가동시간 (Hr) | 현업 직접 입력 | — |
| ③ 생산성 (생산량/일) | **자동 계산** | 생산량 ÷ (가동시간 ÷ 24) |

- **단위는 호기마다 다름** — 제지/화장지 = **톤**, 가공/생리대 = **천개**, 라미네이팅 = **EA**.
  - 화면에는 호기별 단위 그대로 표시(값 환산 없음). 내부 `PowerDB`는 kg/EA로 저장하므로
    표시값 = 내부값 ÷ 1000(톤·천개), ÷ 1(EA). 입·출력 시 `ppToDisp / ppToInternal`로 변환.
- 입력하지 않은 항목(생산량·가동시간)은 **실적값을 그대로 사용**(fallback).

### 핵심 설계
- **전역 상태 `ProdPlan`** (app.js): `ProdPlan.plan[month][line] = { prod?, hours? }`
  - `ProdPlan.get(month, line, field)` → 입력값이 있으면 그 값, 없으면 **실적 fallback**
  - `ProdPlan.productivity(month, line)` → 생산량 ÷ (가동시간 ÷ 24) (표시 단위/일)
  - `localStorage` 영속화 (key: `precost_prodplan_v2`) — 새로고침해도 유지
  - `ProdPlan.bulkSet(rows)` → 엑셀 업로드 일괄 반영, `ProdPlan.clearMonth(month)` → 해당 월 초기화
- **각 모듈은 ProdPlan을 읽기만** 한다. 전력비 `powerRender()`가 `prod`·`hours`를 override로 주입.

### 화면 기능 (`#prodplan`)
- 대상 월 선택 → 호기별 표(호기 / 항목 / 단위 / 실적 / 예상 / 증감), **그룹(제지·화장지·가공·생리대·라미네이팅) 헤더**로 묶음
- 호기당 3행: **생산량(입력)·가동시간(입력)·생산성(자동)**
- 입력 즉시 저장(localStorage) + 생산성·증감%·KPI 실시간 갱신 (입력 포커스 유지)
- `이 달 전체 실적값으로` 버튼으로 월 단위 초기화

### 엑셀 업로드/양식 (SheetJS, CDN `xlsx@0.18.5`)
- **양식 받기**: `호기 | 항목 | 단위 | 2026-01…12` 행렬 다운로드 (실적 생산량·가동시간 채움, SheetJS 없으면 CSV fallback)
  - 생산량은 호기별 **표시 단위**(톤/천개/EA), 가동시간은 Hr 원값
- **엑셀 업로드**: 헤더(`호기`, `항목`, 월 컬럼) 자동 인식 → `항목` 컬럼으로 prod/hours 구분 → `ProdPlan.bulkSet`
  - 월 표기 자동 정규화: `2026-01`, `2026년 1월`, `2026.1`, `2026/01` 등
  - 생산량은 `ppToInternal`로 내부 단위 환산 후 저장, 가동시간은 원값
- 업로드 후 전력비 시뮬레이션에 **즉시 자동 반영**

### 연동 검증 (node)
- 화장지3 2026-02 생산성 = 570톤 ÷ (248/24) = **55.2** (사용자 제공 표와 일치, PASS)
- 제지3 2026-01 실적 원단위 **567.65** / 전력비원단위 **103.55** (PASS)
- 가동시간 **+20%** → 전력사용량 14,712,533 → 17,655,040 kWh(정확히 +20%), 원단위 567.65 → **681.18** (A방식, PASS)

### 기능 진입 경로 (URI)
- `#prodplan` — 예상 생산량 기준정보(입력·엑셀 업로드)
- `#power` — 전력비 시뮬레이션(ProdPlan 자동 반영)
- DEMO_MODE: `?demo=1` 데모 / `?demo=0` 빈 상태(운영)
